# RAG 模式配置说明

## 概述

PaiSmart 系统支持两种 RAG（检索增强生成）模式，用于控制当知识库中检索不到相关内容时，AI 的行为策略。

## 两种模式

### 1. 严格模式 (Strict Mode)

**配置值**: `rag_mode: "strict"`

**行为特点**:
- ✅ **仅使用知识库内容**：AI 只能根据检索到的知识库内容回答问题
- ❌ **禁止使用通用知识**：当检索不到相关内容时，AI 会明确告知用户"知识库中暂无相关信息"
- 🎯 **适用场景**：
  - 企业合规场景（需要确保答案可追溯）
  - 专业领域知识库（避免 AI 幻觉）
  - 法律、医疗等高风险领域
  - 需要严格引用来源的场景

**示例响应**（无检索结果时）：
```
抱歉，知识库中暂无相关信息，无法回答您的问题。
```

---

### 2. 灵活模式 (Flexible Mode)

**配置值**: `rag_mode: "flexible"`

**行为特点**:
- ✅ **优先知识库**：优先使用检索到的知识库内容
- ✅ **允许补充**：当检索不到内容时，AI 可以使用自己的通用知识回答
- 📌 **明确标注**：AI 会在回答开头标注"【注意：以下回答基于AI通用知识，非来自知识库】"
- 🎯 **适用场景**：
  - 通用知识问答
  - 客服场景（需要兜底回答）
  - 混合知识场景（部分来自知识库，部分通用知识）
  - 提升用户体验（避免"无法回答"）

**示例响应**（无检索结果时）：
```
【注意：以下回答基于AI通用知识，非来自知识库】

关于您的问题...[AI 使用通用知识回答]
```

## 配置方法

### 在 `configs/config.yaml` 中设置

```yaml
llm:
  base_url: "https://api.deepseek.com/v1"
  model: deepseek-chat
  api_key: "your-api-key"
  rag_mode: "strict"  # 或 "flexible"
```

### 配置项说明

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `rag_mode` | string | `"strict"` | RAG 模式：`"strict"` 严格模式，`"flexible"` 灵活模式 |

---

## 相关性阈值配置

除了 RAG 模式，系统还支持设置**相关性阈值**，用于过滤低相关性的检索结果。

### 配置示例

```yaml
elasticsearch:
  addresses: "http://127.0.0.1:9200"
  index_name: "knowledge_base"
  min_relevance_score: 0.5  # 最小相关性分数（0-1之间）
```

### 阈值说明

| 阈值范围 | 效果 | 推荐场景 |
|---------|------|---------|
| `0.3-0.4` | 宽松过滤，召回率高 | 灵活模式，希望尽量检索到内容 |
| `0.5-0.6` | **推荐值**，平衡准确率和召回率 | 大多数场景 |
| `0.7-0.9` | 严格过滤，只保留高相关性内容 | 严格模式，要求高精度 |

**注意**：
- 分数 < 阈值的检索结果会被过滤掉
- 如果所有结果都被过滤，系统会触发"无检索结果"流程
- 阈值设置过高可能导致漏检，过低可能引入噪声

## 元数据响应

系统在 WebSocket 响应中会发送元数据消息，告知前端当前的检索状态和模式：

```json
{
  "type": "metadata",
  "has_knowledge": false,
  "result_count": 0,
  "rag_mode": "strict",
  "message": "知识库中未找到相关内容，AI 将提示无法回答"
}
```

**字段说明**：
- `has_knowledge`: 是否检索到知识库内容
- `result_count`: 检索到的结果数量
- `rag_mode`: 当前 RAG 模式
- `message`: 给用户的提示信息

## 模式对比表

| 特性 | 严格模式 (Strict) | 灵活模式 (Flexible) |
|------|------------------|-------------------|
| 使用知识库内容 | ✅ 是 | ✅ 是（优先） |
| 使用通用知识 | ❌ 否 | ✅ 是（标注） |
| 无结果时行为 | 拒绝回答 | 使用通用知识 |
| 答案可追溯性 | ✅ 高 | ⚠️ 中（需查看标注） |
| 用户体验 | 专业严谨 | 友好灵活 |
| 幻觉风险 | ✅ 低 | ⚠️ 中（灵活模式下存在） |

## 最佳实践

### 1. 选择合适的模式

- **企业内部知识库**：推荐 `strict` 模式
- **客服机器人**：推荐 `flexible` 模式
- **混合场景**：根据业务需求动态调整

### 2. 调整相关性阈值

```yaml
# 严格模式配置示例
llm:
  rag_mode: "strict"
elasticsearch:
  min_relevance_score: 0.7  # 高阈值，确保精准

# 灵活模式配置示例
llm:
  rag_mode: "flexible"
elasticsearch:
  min_relevance_score: 0.4  # 低阈值，提高召回
```

### 3. 监控和调优

- 查看日志中的 `[ChatService]` 标记，了解检索情况
- 根据用户反馈调整阈值
- 定期评估 AI 回答质量

## 技术实现细节

### 系统提示词切换

系统根据 `rag_mode` 配置自动选择对应的提示词：

- **严格模式**：使用 `llm.prompt.rules_strict`
- **灵活模式**：使用 `llm.prompt.rules_flexible`

### 代码位置

- 配置结构：`internal/config/config.go`
- RAG 逻辑：`internal/service/chat_service.go`
- 检索过滤：`internal/service/search_service.go`

## 常见问题

### Q1: 如何切换模式？

修改 `configs/config.yaml` 中的 `llm.rag_mode` 字段，重启服务即可。

### Q2: 能否动态切换模式？

当前版本需要重启服务。如需运行时切换，可以通过配置中心或环境变量实现。

### Q3: 灵活模式下如何确保答案质量？

- 设置合适的相关性阈值（推荐 0.5+）
- 监控 AI 回答，发现问题及时优化知识库
- 在前端显示"基于AI通用知识"的标注

### Q4: 严格模式下用户体验差怎么办？

- 优化知识库内容的覆盖度
- 降低相关性阈值（如 0.4）
- 提供更友好的"无法回答"提示（在 `no_result_text` 中配置）
